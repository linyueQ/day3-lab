## 1. 概念模型

```
Session（会话）1 ── * QARecord（问答记录）
   │                      │
   │  session_id (PK)     │  id (PK)
   │  title               │  session_id (FK → Session)
   │  query_count         │  query / answer / file_info
   │  created_at          │  llm_used / response_time_ms
   │  updated_at          │  answer_source / timestamp
   └  status              └  file_path

ReportFile（研报文件）1 ── * QARecord
   │
   │  file_id (PK)
   │  session_id (FK)
   │  file_name
   │  file_size
   │  file_type
   │  parse_status
   │  created_at
```

> **核心关系**：
> - 一个 Session 可包含零到多条 QARecord。外键通过 `session_id` 字段逻辑引用。
> - 一个 ReportFile 可被多个 QARecord 引用（同一文件多次提问）。

## 2. 存储引擎

| 项 | 说明 |
|----|------|
| **前端数据** | React State（非持久化），页面刷新后清空 |
| **后端文件** | 上传的研报文件存储在 `uploads/` 目录 |
| **后端配置** | API Key 等敏感配置通过 `.env` 环境变量管理 |
| 持久化策略 | 对话数据不做持久化，仅在页面生命周期内有效 |

> **注**：当前阶段不使用数据库或 LocalStorage。以下实体定义描述的是 React State 中的数据结构和后端 API 响应格式。

## 3. 实体定义 — Session

| 字段 | 类型 | 必填 | 约束 | 说明 |
|------|------|------|------|------|
| `session_id` | `string` (UUID) | 是 | **PK** | 会话唯一标识 |
| `title` | `string` | 是 | ≤50 字符 | 默认"新会话"，首次上传文件后自动重命名 |
| `created_at` | `string` (ISO-8601) | 是 | UTC+Z | 创建时间 |
| `updated_at` | `string` (ISO-8601) | 是 | UTC+Z | 最后更新时间 |
| `query_count` | `integer` | 是 | ≥ 0 | 累计问答次数 |
| `status` | `string` | 是 | 枚举 | `active`\|`archived`\|`deleted` |

## 4. 实体定义 — QARecord

| 字段 | 类型 | 必填 | 约束 | 说明 |
|------|------|------|------|------|
| `id` | `string` | 是 | **PK**，`rec_{ts}` | 基于时间戳生成 |
| `session_id` | `string` | 是 | **FK** → Session | 所属会话 |
| `file_id` | `string` | 否 | **FK** → ReportFile | 关联研报文件 |
| `query` | `string` | 是 | 1–500 字符 | 用户提问原文 |
| `answer` | `string` | 是 | — | LLM/Demo 返回的答案（结构化 JSON） |
| `llm_used` | `boolean` | 是 | — | 是否使用真实 LLM |
| `model` | `string\|null` | 是 | — | 模型标识 |
| `response_time_ms` | `integer` | 是 | SLO < 5000 | 响应耗时（ms） |
| `answer_source` | `string\|null` | 否 | 枚举 | `copaw`\|`bailian`\|`demo` |
| `timestamp` | `string` (ISO-8601) | 是 | UTC+Z | 记录创建时间 |

## 5. 实体定义 — ReportFile

| 字段 | 类型 | 必填 | 约束 | 说明 |
|------|------|------|------|------|
| `file_id` | `string` (UUID) | 是 | **PK** | 文件唯一标识 |
| `session_id` | `string` | 是 | **FK** → Session | 所属会话 |
| `file_name` | `string` | 是 | ≤255 字符 | 原始文件名 |
| `file_size` | `integer` | 是 | ≤104857600 (100MB) | 文件大小（字节） |
| `file_type` | `string` | 是 | 枚举 | `pdf`\|`doc`\|`docx` |
| `file_path` | `string` | 是 | — | 服务器存储路径 |
| `parse_status` | `string` | 是 | 枚举 | `pending`\|`parsing`\|`completed`\|`failed` |
| `parse_progress` | `integer` | 否 | 0-100 | 解析进度百分比 |
| `parse_result` | `object` | 否 | — | 解析结果（标题、评级、核心观点等） |
| `created_at` | `string` (ISO-8601) | 是 | UTC+Z | 上传时间 |
| `updated_at` | `string` (ISO-8601) | 是 | UTC+Z | 最后更新时间 |

## 6. 后端服务方法清单

> 会话和问答数据由前端 React State 管理，后端仅提供 API 端点供前端调用。以下列出后端核心服务方法。

### 6.1 LLM 管理

| 方法 | 签名 | 行为 | 关联 TC |
|------|------|------|---------|
| `get_providers` | `() → list[dict]` | 读取 `.env` 配置，返回可用 LLM Provider 列表（不含 API Key） | TC-006-01 |
| `get_default_provider` | `() → str` | 返回默认 Provider ID | TC-006-02 |

### 6.2 问答服务

| 方法 | 签名 | 行为 | 关联 TC |
|------|------|------|---------|
| `ask` | `(query: str, session_id: str, file_content: str\|None, provider: str\|None) → dict` | 调用指定/默认 LLM Provider，返回结构化回答，支持降级 | TC-002-02 |

### 6.3 研报文件管理

| 方法 | 签名 | 行为 | 关联 TC |
|------|------|------|---------|
| `save_file` | `(session_id: str, file: FileStorage) → dict` | 保存上传文件到 `uploads/` 目录，返回文件元数据 | TC-002-01 |
| `parse_file` | `(file_id: str) → dict` | 解析文件，更新解析状态和进度 | TC-002-01 |
| `get_parse_status` | `(file_id: str) → dict` | 获取文件解析状态和结果 | TC-002-06 |

### 6.4 深度分析

| 方法 | 签名 | 行为 | 关联 TC |
|------|------|------|---------|
| `trigger_analysis` | `(session_id: str, file_id: str) → dict` | 调用 Rabyte Skill API 触发深度分析，返回 analyze_id | TC-005-01 |
| `get_analysis_status` | `(analyze_id: str) → dict` | 查询深度分析状态和结果 | TC-005-02 |

## 7. 关键业务逻辑

**示例 — 首次上传自动命名**：

```python
if session["query_count"] == 0 and file_info:
    # 使用文件名前20字作为会话标题
    session["title"] = file_info["file_name"][:20] + ("..." if len(file_info["file_name"]) > 20 else "")
```

> **规则**：首次上传文件时，自动使用文件名（截断至20字）作为会话标题。

| 逻辑 | 触发条件 | 行为 |
|------|----------|------|
| 首次上传自动命名 | `query_count` == 0 且上传文件 | 标题 = 文件名前 20 字 + `...` |
| 解析进度更新 | 文件解析过程中 | 实时更新 `parse_progress` 字段 |
| 文件大小校验 | 上传文件时 | 校验 file_size ≤ 104857600 (100MB)，超限拒绝 |
| 解析超时处理 | 解析超过 30s | parse_status 设为 `failed`，记录错误信息 |
| LLM Provider 切换 | 用户在前端选择不同模型 | 后续 /ask 请求携带 provider 参数 |
| 深度分析触发 | 用户点击"深度分析"按钮 | 调用 Skill API，异步返回结果 |

## 8. API 响应 DTO 概要

> 详细端点定义见 `09`。此处列出关键 DTO 与存储字段的映射关系。

| API 端点 | 核心 DTO 字段 | 额外注入 |
|----------|---------------|----------|
| `POST /sessions` | session_id, title, created_at, query_count, status | `traceId` |
| `GET /sessions` | sessions[] | `traceId` |
| `GET /sessions/{id}` | session 详情 + 关联 files[] | `traceId` |
| `POST /files/upload` | file_id, file_name, parse_status, created_at | `traceId` |
| `GET /files/{id}/status` | file_id, parse_status, parse_progress, parse_result | `traceId` |
| `POST /ask` | answer, llm_used, model, response_time_ms, answer_source | `traceId` |
| `GET /sessions/{id}/records` | records[]（含 query, answer, timestamp） | `traceId` |
| `GET /llm/providers` | providers[]（id, name, model, available）, default | `traceId` |
| `POST /analyze` | analyze_id, status | `traceId` |
| `GET /analyze/{id}/status` | analyze_id, status, progress, result | `traceId` |
