> **本文只做一件事**：按用户能理解的子场景拆成多条 **User Story**，并为每条写清 **REQ-ID**、**验收标准（AC）**。

## 字段说明 — 每条 UserStory 必须包含以下字段

| 字段 | 填写说明 | 数据来源 ← | 流向下游 → |
|------|----------|------------|-----------|
| **需求编号** | 唯一编号，如 REQ-M5RA-001 | ← `02` 采集编号 | → `14` 追踪矩阵首列 |
| **故事** | 作为【角色】，我希望【行为】，以便【价值】 | ← `04` PRD 场景拆解 | → `06` FSD 功能点 |
| **优先级** | P0（必须）/ P1（应该）/ P2（可选） | ← `03` §4 成功标准 | → `12` 里程碑排期 |
| **来源** | 对应 03/04 哪一条需求 | ← `03` 范围 · `04` 场景 | |
| **前置条件** | 触发该故事的前提状态 | ← `04` 用户流程 | → `13` 测试前置 |
| **功能对齐** | 对应功能规格 + 非功能约束 | | → `06` FSD · `07` NFR |
| **设计对齐** | 对应架构、接口、数据、安全 | | → `08` 架构 · `09` API · `10` Data · `11` 安全 |
| **验证对齐** | 对应测试用例 + 追踪矩阵行 | | → `13` TC · `14` 追踪矩阵行 |
| **验收标准** | AC，每条可独立验证 | ← 故事 + `07` NFR | → `13` 测试用例依据 |

| **来源** | ← 04 §5.1 投资控股型平台公司信用资质分析范式 / 04 §5.4 研报分析两阶段工作流 |

## US-006（P1）前端大模型切换

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-M5RA-006 |
| **故事** | 作为投研分析师，我希望在前端切换不同的大模型引擎，以便根据分析需求选择最适合的 LLM |
| **优先级** | P1 |
| **来源** | ← 04 §5.2 大模型切换 |
| **前置条件** | 后端已配置至少一个 LLM Provider 的 API Key |
| **功能对齐** | → 06 §模型切换组件 · 07 §LLM 安全 |
| **设计对齐** | → 08 §LLM 管理 · 09 GET /api/llm/providers · 11 §密钥安全 |
| **验证对齐** | → 13 TC-006 · 14 第6行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-006-01 | 前端 Header 区域显示当前模型名称和切换按钮 |
| AC-006-02 | 点击切换按钮展示后端已配置的可用模型列表（通过 GET /api/llm/providers 获取） |
| AC-006-03 | 选择目标模型后，后续对话自动使用新模型 |
| AC-006-04 | API Key 在后端 .env / 环境变量中统一管理，前端不涉及 Key 输入 |
| AC-006-05 | 模型列表接口不返回 API Key 明文，仅返回 provider id、名称、可用状态 |
