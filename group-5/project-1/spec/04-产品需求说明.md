## 5.1 投资控股型平台公司信用资质分析范式

> 本功能支持对上传的研报进行结构化深度分析，基于"三层穿透、八维分析"框架。

### 5.1.1 三层穿透分析框架

| 层级 | 分析对象 | 核心关注点 |
|------|----------|------------|
| 第一层 | 母公司 | 资产质量、债务压力、现金流来源（是否依赖子公司分红/股东支持） |
| 第二层 | 核心子公司 | 经营状况、盈利能力、是否为优质资产 |
| 第三层 | 股东 | 股东在资产注入、资金拆借、增信等方面的支持意愿和能力 |

### 5.1.2 八个分析维度

| 维度 | 分析内容 | 关键指标 |
|------|----------|----------|
| 维度一 | 战略定位与股东背景 | 股权结构、战略定位、历史沿革 |
| 维度二 | 业务结构拆解 | 现金流贡献板块、亏损拖累板块、风险积累板块 |
| 维度三 | 母公司口径财务分析 | 长期股权投资、有息债务、取得投资收益收到的现金 |
| 维度四 | 合并口径财务分析 | 资产质量、盈利能力、现金流质量 |
| 维度五 | 资金流向与勾稽关系 | 资金来源端、资金占用端、勾稽验证 |
| 维度六 | 再融资能力分析 | 银行授信、债券市场认可度、股东借款稳定性 |
| 维度七 | 外部支持分析 | 股东支持历史、集团实力、内部协同 |
| 维度八 | 评级一致性比对 | 可比主体选择、比对维度、比对结论 |

### 5.1.3 分析输出结构

分析结果按以下结构输出：

1. **核心要点总结**：切忌迷信合并报表、穿透业务看本质、追踪资金流向、量化股东支持
2. **分层偿债来源评估**：
   - 第一偿债来源：再融资能力
   - 第二偿债来源：资产变现/盘活能力
   - 第三偿债来源：自身经营现金流
   - 最终支撑：股东支持

## 5.2 大模型切换

> API Key 统一在后端配置管理，前端仅提供大模型切换按钮，用户可一键切换不同 LLM 引擎。

### 5.2.1 功能描述

| 功能项 | 描述 |
|--------|------|
| 前端交互 | 提供"切换大模型"按钮/下拉选择器，展示可用模型列表 |
| 支持的模型 | OpenAI GPT 系列、阿里云百炼、DeepSeek、智谱 AI 等 |
| API Key 管理 | 所有 API Key 在**后端配置文件/环境变量**中统一管理，前端不涉及 Key 输入 |
| 模型列表来源 | 前端通过接口从后端获取当前已配置且可用的模型列表 |

### 5.2.2 后端配置方式

| 项 | 说明 |
|------|------|
| 配置位置 | 后端 `.env` 文件或环境变量 |
| 配置格式 | `LLM_PROVIDER_<NAME>_API_KEY=xxx`，支持多个 Provider 并存 |
| 安全要求 | API Key 不落日志，不通过接口暴露给前端 |
| 热更新 | 修改配置后重启服务即可生效，无需重新部署 |

```bash
# .env 配置示例
LLM_PROVIDER_BAILIAN_API_KEY=sk-xxxxxx
LLM_PROVIDER_BAILIAN_MODEL=qwen-max
LLM_PROVIDER_DEEPSEEK_API_KEY=sk-xxxxxx
LLM_PROVIDER_DEEPSEEK_MODEL=deepseek-chat
LLM_DEFAULT_PROVIDER=bailian
```

### 5.2.3 前端交互流程

1. 页面加载时调用 `GET /api/llm/providers` 获取可用模型列表
2. 前端 Header 区域显示当前模型名称 + 切换按钮
3. 用户点击切换按钮 → 弹出下拉列表展示所有可用模型
4. 选择目标模型后，后续对话自动使用新模型
5. 用户选择的模型偏好保存在 LocalStorage，刷新页面后保持

### 5.2.4 接口设计

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/llm/providers` | GET | 返回后端已配置的可用模型列表（不含 API Key） |

**响应示例**：
```json
{
  "providers": [
    { "id": "bailian", "name": "阿里云百炼", "model": "qwen-max", "available": true },
    { "id": "deepseek", "name": "DeepSeek", "model": "deepseek-chat", "available": true }
  ],
  "default": "bailian"
}
```

## 5.4 研报分析两阶段工作流

> 研报分析分为两个阶段：**阶段一**基于预设 prompt 模板与 LLM 进行结构化问答；**阶段二**调用开源 Skill 工作流进行深度分析。两个阶段顺序执行，阶段二由用户主动触发。

### 5.4.1 阶段一：基于 Prompt 模板的结构化问答

> 用户上传研报后，系统基于预设的信用分析 prompt 模板，将用户提问与研报内容一起发送给 LLM，返回结构化回答。

| 项 | 说明 |
|------|------|
| 触发方式 | 用户输入问题后**自动执行** |
| 分析引擎 | 百炼 / 用户在前端切换选择的 LLM（API Key 后端统一管理） |
| 输入 | 系统 prompt 模板 + 研报文本上下文 + 用户问题 |
| 输出 | 结构化回答（含核心观点、关键数据对比、评级信息等） |
| 对话模式 | 支持**多轮对话**，系统保持上下文连续性 |
| 响应方式 | 流式输出，实时显示回答内容 |
| Prompt 模板 | 基于 §5.1 信用资质分析范式（三层穿透、八维分析）构建，确保回答结构化、可比较 |

**阶段一核心需求**：

| 编号 | 需求 | 优先级 |
|------|------|--------|
| P1-01 | 系统内置信用分析 prompt 模板，引导 LLM 按八维框架输出 | P0 |
| P1-02 | 支持多轮对话，每轮问答自动携带历史上下文 | P0 |
| P1-03 | 流式响应（SSE），前端逐字渲染回答 | P0 |
| P1-04 | 回答结果自动保存至前端状态，可在会话管理中查看历史对话 | P0 |
| P1-05 | 用户可通过前端按钮切换 LLM 引擎，切换后新对话生效 | P1 |

### 5.4.2 阶段二：深度研报分析（Skill 工作流）

> 阶段一完成后，用户可点击前端"深度分析"按钮，触发 Rabyte 开源 Skill 对研报进行全面深度分析。该阶段为**独立工作流**，异步执行，分析结果独立于问答记录展示。

| 项 | 说明 |
|------|------|
| 触发方式 | 用户点击**"深度分析"按钮**手动触发（非自动） |
| 前置条件 | 阶段一至少完成一轮问答（确保研报已解析成功） |
| 分析引擎 | Rabyte Skill API（开源研报分析 Skill） |
| 输入 | 已上传的研报原始文件 |
| 输出 | 完整研报分析报告（表格提取、图表识别、关键信息抽取、综合评估） |
| 执行方式 | 异步执行，前端显示进度状态（排队中 → 分析中 → 已完成） |
| 结果展示 | 独立的"分析报告"面板，支持 Markdown 渲染 |

**阶段二核心需求**：

| 编号 | 需求 | 优先级 |
|------|------|--------|
| P2-01 | 前端对话区域下方展示"深度分析"按钮，仅在研报上传成功后可点击 | P1 |
| P2-02 | 点击后调用 Rabyte Skill API，传入研报文件，异步等待结果 | P1 |
| P2-03 | 分析过程中展示进度状态：排队中 → 分析中 → 已完成 / 失败 | P1 |
| P2-04 | 分析完成后在右侧/下方独立面板展示完整分析报告 | P1 |
| P2-05 | 分析报告支持 Markdown 渲染，含表格、图表等结构化内容 | P1 |
| P2-06 | 分析报告保存至前端状态，可在会话管理中再次查看 | P1 |
| P2-07 | 每个会话仅允许触发一次深度分析（已完成/进行中时按钮置灰） | P2 |
| P2-08 | Skill API 失败时降级提示"深度分析暂不可用"，不影响阶段一 | P1 |

### 5.4.3 两阶段关系与交互设计

**阶段关系**：

| 关系 | 说明 |
|------|------|
| 顺序依赖 | 阶段二依赖阶段一（研报已上传解析），但不依赖具体问答内容 |
| 独立运行 | 两阶段使用不同引擎（LLM vs Skill API），结果互不影响 |
| 互补定位 | 阶段一侧重用户交互式问答，阶段二侧重一次性全面分析 |
| 结果合并 | 两阶段结果均归属同一会话，在会话详情中分区展示 |

## 5.5 数据存储策略

> 数据暂不需要持久化，对话内容保存在前端运行时内存（React State / Context）中，支持用户在会话管理中查看历史对话。页面刷新后数据清空。

### 5.5.1 存储方案

| 数据类型 | 存储位置 | 存储方式 | 生命周期 |
|----------|----------|----------|----------|
| 会话列表 | 前端内存（React State） | Array | 页面生命周期内有效，刷新后清空 |
| 问答记录 | 前端内存（React State） | Array（按 session_id 分组） | 页面生命周期内有效，刷新后清空 |
| 当前选择的模型 | 前端内存（React State） | String | 页面生命周期内有效，刷新后恢复默认 |
| API Key 配置 | 后端 `.env` / 环境变量 | 明文（服务端安全） | 后端持久化 |
| 上传文件 | 前端内存 | File 对象 | 会话级保留，页面刷新后清空 |
| 分析报告 | 前端内存（React State） | String（Markdown） | 页面生命周期内有效，刷新后清空 |

## 6. 验收与成功标准（PRD 层）

> 从 `03` §4 可度量标准出发，加上演示条件。
> **参考句式**："种子数据 ≥ N 条下，XXX 在 N 分钟内完成"

| 标准 | 度量 |
|------|------|
| 可演示 | 种子数据 ≥ 10 条下，主路径 + 历史 + 能力状态 **3 分钟内**演示完毕 |
| 文件解析性能 | 单文件 ≤ 100MB，解析时间 ≤ 5s |
| 数据提取准确率 | 关键信息提取准确率 ≥ 95% |
