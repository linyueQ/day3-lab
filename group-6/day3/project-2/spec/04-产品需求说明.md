# 04 — 产品需求说明（PRD）

---

| 项 | 值 |
|---|---|
| 模块编号 | M2-IRA |
| 模块名称 | 投研分析平台（AI 研报助手） |
| 文档版本 | v0.1 |
| 阶段 | Proposal（Why → What 过渡） |
| 上游 | `03` 立项提案（已通过） |
| 下游 | → `05` UserStory 与验收标准 |

---

> **04 的定位**：`03` 说了"为什么做"，04 翻译成"做什么、不做什么、多重要"。`05` 再拆成用户故事和验收标准。

## 1. 愿景与目标

| 03 痛点 | 04 目标 |
|---------|---------|
| 研报阅读耗时，关键信息提取不完整 | 上传 PDF 研报后 LLM 自动提取标题、评级、目标价、核心观点、股票代码/名称，一键获取结构化摘要 |
| 多份研报对比困难，需手动整理 | 选择同一公司多份研报，系统自动输出相似合并 + 差异高亮的对比视图 |
| 已读研报无法高效检索和复用 | 以股票为维度建立知识库，自动汇总最近观点 |
| 股票数据分散在多个系统 | 知识库页面集成 AKShare 行情数据（PE/PB/市值等），无需切换系统 |

## 2. 目标用户

| 角色 | 使用目标 | 频率 |
|------|----------|------|
| 投研分析师 | 快速阅读研报、提取关键信息、对比不同券商观点 | 每天多次 |
| 高级研究员 | 基于知识库进行深度分析、多维度研报比对 | 每天多次 |
| 研究总监 | 快速了解团队覆盖股票的最新研报观点汇总 | 每天 1-2 次 |

## 3. 核心业务场景

| 编号 | 场景名 | 用户路径 |
|------|--------|----------|
| SC-01 | 上传并解析研报 | 用户上传 PDF 研报 → 系统自动提取文本 → LLM 解析结构化字段（标题、评级、目标价、核心观点、股票代码/名称）→ 自动打标签 → 存入知识库 |
| SC-02 | 浏览知识库 | 用户选择某只股票 → 查看该股票关联的所有研报 → 查看最近观点汇总 → 查看行情数据（PE/PB/市值）→ 按时间筛选/排序 |
| SC-03 | 研报比对 | 用户选择同一公司的 2+ 份研报 → 系统执行比对 → 输出相似信息合并视图 + 差异高亮标注 |
| SC-04 | 研报管理 | 用户查看已上传研报列表 → 按股票代码/行业/时间筛选 → 查看研报详情 → 删除研报 |

## 4. 产品规则

| 编号 | 规则 | 来源 |
|------|------|------|
| R-01 | 上传的研报 MUST 为 PDF 格式，单文件大小 ≤ 50MB | 03 §3 范围边界 |
| R-02 | LLM 解析 MUST 提取 6 个核心字段：标题、评级、目标价、核心观点、股票代码、股票名称 | 03 §4 成功标准 |
| R-03 | 知识库 MUST 以股票代码为聚合维度，每个股票页面展示关联研报、最近观点汇总、行情数据 | 需求总结 §三 |
| R-04 | 研报比对 MUST 支持 ≥ 2 份同一公司研报，输出相似合并 + 差异高亮 | 03 §4 成功标准 |
| R-05 | 行情数据通过 AKShare API 获取，API 不可用时降级显示"暂无数据" | 03 §5 风险 |
| R-06 | 研报文件 MUST 存储在后端服务器本地，不上传到外部云存储 | 03 §3 范围边界 |

## 5. 能力与优先级

| 子能力 | 优先级 | 对应场景 |
|--------|--------|----------|
| 研报上传与 PDF 文本提取 | **P0** | SC-01 |
| LLM 自动解析（提取 6 个核心字段 + 打标签） | **P0** | SC-01 |
| 知识库管理（按股票聚合、时间筛选、观点汇总） | **P0** | SC-02 |
| 研报比对（相似合并 + 差异高亮） | **P0** | SC-03 |
| 研报管理（列表、筛选、删除） | **P0** | SC-04 |
| AKShare 行情数据集成（PE/PB/市值） | **P1** | SC-02 |

## 6. 验收与成功标准（PRD 层）

| 标准 | 度量 |
|------|------|
| 解析完整 | 上传 PDF 研报后，系统自动解析出标题、评级、目标价、核心观点、股票代码、股票名称 6 个字段并持久化 |
| 比对可用 | 输入同一公司 ≥ 2 份研报，输出包含相似合并和差异高亮的比对视图 |
| 知识库聚合 | 按股票维度展示关联研报 + 最近观点汇总 + 行情数据 |

## 7. 与 05 用户故事的关系

> **04 定"做什么、多重要"；05 定"谁要什么、怎么验收"。** 填写完本文后，下一步进入 `05` 将场景拆成 US + AC。

| 本文（PRD） | 作用 | 05 中落地 |
|-------------|------|-----------|
| §3 核心场景 | 一条叙事场景 → 一条或多条 US | SC-01 → US-001 + US-002 |
| §4 产品规则 | 字段/状态/错误口径 | 拆入各 US 的 AC |
| §5 能力优先级 | P0 / P1 | 与各 US 优先级一致 |
