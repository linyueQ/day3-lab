## 1. 背景与问题

> 投研团队在日常工作中面临以下核心痛点：

| 痛点 | 来源 |
|------|------|
| 研报阅读耗时，每天 2-3 小时，关键信息（评级、目标价、核心观点）提取不完整 | S-01 会议纪要、S-02 邮件 |
| 不同券商对同一公司的研报比对困难，需手动复制粘贴到 Excel | S-01 会议纪要、S-03 用户访谈 |
| 已读研报内容无法高效检索和复用，缺乏以股票为维度的知识沉淀 | S-02 邮件、S-03 用户访谈 |
| 股票基本面数据分散在多个系统，查询不便 | S-01 会议纪要、S-02 邮件 |

## 2. 提案概要

构建一个 AI 驱动的投研分析平台，支持研报上传与自动解析（LLM 提取标题、评级、目标价、核心观点、股票代码/名称），以股票为维度聚合建立知识库，提供研报智能比对（相似合并 + 差异高亮）。核心价值在于用 LLM 大幅降低研报阅读和比对的人力成本。

## 3. 范围边界

| 做（范围内） | 不做（范围外） |
|-------------|---------------|
| 研报管理：上传 PDF 格式研报，自动打标签（股票代码、行业），按标签分类管理 | 非 PDF 格式研报解析（Word、HTML 等） |
| 研报解析：LLM 自动提取标题、评级、目标价、核心观点、股票代码/名称 | 与核心字段无关的内容提取 |
| 知识库：以股票为聚合维度，支持时间筛选/排序，展示行情数据，自动输出最近观点汇总 | 舆情监控 |
| 研报比对：同一上市公司多份研报的相似合并 + 差异高亮 | 跨公司/跨行业对比 |
| 股票数据：通过 AKShare API 获取基本资料、行情、财务指标 | 实时行情推送 |
| 文件存储：后端服务器本地存储 | 云存储 |

## 4. 成功标准（可度量）

| 标准 | 度量方式 |
|------|----------|
| 研报解析完整 | 输入 PDF 研报，自动解析出标题、评级、目标价、核心观点、股票代码、股票名称 6 个核心字段，并保存到内存数据库 |
| 研报比对可用 | 输入同一公司 2 份及以上研报，系统输出相似信息合并视图 + 差异高亮标注 |
| 知识库聚合 | 以股票维度展示关联研报、最近观点汇总、行情数据（PE/PB/市值） |

## 5. 风险（简表）

| 风险 | 影响 | 应对 |
|------|------|------|
| LLM 解析准确率不足 | 提取的评级、目标价等字段可能有误 | LLM Prompt 工程优化 + 人工校验入口 |
| PDF 格式多样性 | 部分券商研报 PDF 结构特殊导致文本提取失败 | 多种 PDF 解析库兜底（PyPDF2 → pdfplumber） |
| AKShare API 不稳定 | 股票行情数据获取失败 | 本地缓存 + 降级显示"暂无数据" |
| 知识库数据量增长 | 内存数据库性能下降 | 初期内存足够，后续可迁移到 SQLite/PostgreSQL |

## 6. 立项前置条件（Go / No-Go）

- [ ] 后端与前端可独立启动
- [ ] 有可用 LLM API（DashScope / OpenAI 兼容接口）
- [ ] AKShare 库可正常调用
- [ ] PDF 解析库可正常提取文本
- [ ] 演示用研报 PDF 文件准备就绪（≥ 2 份同一公司）
