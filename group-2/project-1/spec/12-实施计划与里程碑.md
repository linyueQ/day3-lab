## 1. 里程碑（S1～S6）

> **切分原则**：每刀切出一个**可演示的用户价值**。不是"接口写好了"，而是"能给业务看什么"。

| 阶段 | 交付要点（可演示） | 满足的 REQ/US | 验证 TC |
|------|-------------------|---------------|---------|
| **S1** 会话管理 | 会话 CRUD API + 前端侧栏（新建/列表/删除），主区域三态渲染 | REQ-M1QA-001 / US-001 | TC-M01-041~043, TC-M01-046 |
| **S2** 问答核心 | 问答提交 API（三级降级）+ 输入区 + 对话历史展示，支持 Markdown 流式渲染 | REQ-M1QA-002 / US-002, US-004 | TC-M01-001~004, TC-M01-044~045 |
| **S3** 溯源与健康 | 原文溯源（引用标记点击→PDF预览定位高亮）+ 健康检查接口 + 能力状态芯片 | REQ-M1QA-003, REQ-M1QA-005 / US-003, US-005 | TC-M01-003, TC-M01-048, TC-M01-005 |
| **S5** 研报对比 | 研报列表 + 多研报对比（财务/观点/风险三标签页），Header 视图切换 | REQ-M1QA-006 / US-006 | TC-M01-061~063 |
| **S6** 舆情监控 | RSS 多源聚合 + 优先级评分 + 5 分钟缓存 + 强制刷新，舆情监控视图 | REQ-M1QA-007 / US-007 | TC-M01-071~074 |
| **S4** 发布准备 | 全链路集成测试 + 性能调优 + 文档完善 + 种子数据准备 | 全部 REQ | 全部 TC |

> **注**：S5/S6 编号保留逻辑顺序，实际可与 S3 并行推进。

## 2. WBS 任务拆分

> **拆分原则**：按**工程职责**切（路由、存储、Agent、前端），不按功能切。一个 W 可跨多个 S 迭代推进。

| 任务ID | 任务名称 | 具体内容 | DoD（指向 Spec） | 推进阶段 |
|--------|----------|----------|------------------|----------|
| W1 | 路由实现 | `server.js` 11 个端点注册（Express 主后端）、参数校验、错误码映射、traceId 注入；`agent_bp.py` Flask 备选后端 | 对齐 `09` 全部 11 个端点与错误码，响应格式符合 `09` 规格 | S1 起贯穿 S4 |
| W2 | 存储层 | `storage.py` JSON CRUD，Session/QARecord/DocChunk 三实体；`server.js` 内置 JSON 存储（sessions/records） | 对齐 `10` 数据模型，Session 含 last_query 字段，通过全部 Storage TC | S1～S2 |
| W3 | Agent 编排 | `agent.py` 三级降级逻辑，`copaw_bridge.py` / `bailian_qa.py` 提供商实现 | 对齐 `08` §4 三级降级编排，降级静默执行，Demo 兜底 | S2～S3 |
| W4 | 前端视图 | React 19 组件：Sidebar / ChatArea / DocPreview / Header / ReportCompare / SentimentMonitor，自定义 CSS 变量体系 | 对齐 `06` FSD 三视图切换（chat/compare/sentiment） + 各组件交互规格，通过 E2E 测试 | S1 起贯穿 S6 |
| W5 | SSE 流式 | `09` §3 POST /ask SSE 流式响应实现，前端 EventSource 消费 | 首字响应 <3s，流式打字机效果，Markdown 实时解析 | S2 |
| W6 | 测试门禁 | pytest 单元/集成测试（Flask），Jest/手动测试（Express），覆盖率 ≥80% | 对齐 `13` 质量门禁，P0 TC 全绿 | S2 起贯穿 S6 |
| W7 | 研报对比 | `server.js` GET /reports + POST /reports/compare 端点；`parseReport` Markdown 解析提取多维度数据；前端 ReportCompare.jsx 三标签页对比 | 对齐 `09` §11-§12，`06` §8 研报对比视图规格 | S5 |
| W8 | 舆情监控 | `server.js` GET /news 端点；rss-parser 多源 RSS 聚合；5 分钟缓存 + forceRefresh；优先级评分；前端 SentimentMonitor.jsx | 对齐 `09` §13，`06` §9 舆情监控视图规格 | S6 |

## 5. 依赖与风险

**依赖链**：

**S1 会话管理** → **S2 问答核心**（必须先有会话才能提问）→ **S3 溯源与健康**（需要问答记录才能溯源）→ **S4 发布准备**

**S5 研报对比** 和 **S6 舆情监控** 可与 S3 并行，仅依赖 S1 的基础框架（路由 + 前端视图切换）。

| 风险 | 影响 | 应对 |
|------|------|------|
| 上游 LLM 不可用 | 问答降级至 Demo 模式 | 三级降级兜底，S2 完成前验证 Demo 模式 |
| 非标格式研报解析错乱 | 研报对比维度提取不完整 | parseReport 按固定标题匹配，无法提取时返回空值并提示 |
| SSE 流式实现复杂度超预期 | S2 延期 | 预留 2 天 buffer，复杂场景降级为非流式 |
| RSS 源不可达或返回格式异常 | 舆情监控数据缺失 | 5 分钟缓存兜底 + 单源失败不影响其他源 + sources[] 返回各源状态 |
| Express 主后端与 Flask 备选后端 API 不一致 | 切换后行为差异 | 以 Express 为主，Flask 仅保留核心问答端点 |
