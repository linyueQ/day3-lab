## 1. 概念模型

```
Session（会话）1 ── * QARecord（问答记录）
   │                      │
   │  session_id (PK)     │  record_id (PK)
   │  title               │  session_id (FK → Session)
   │  query_count          │  query / answer / llm_used
   │  last_query           │  response_time_ms / answer_source
   │  created_at           │  model / timestamp / references
   └  updated_at

DocChunk（文档块）          Report（研报，运行时解析）
   │                          │
   │  chunk_id (PK)           │  filename (PK)
   │  doc_title / doc_url     │  companyName / stockCode
   │  page                    │  rating / targetPrice / currentPrice
   │  highlight_text / bbox   │  financials / coreViews / risks

News（新闻，运行时聚合）
   │
   │  id (PK, MD5 hash)
   │  title / summary / source
   │  publishedAt / tags / priority
```

> **核心关系**：一个 Session 可包含零到多条 QARecord。外键通过 `session_id` 字段逻辑引用。DocChunk 为独立实体，QARecord.references 通过 `chunk_id` 逻辑引用 DocChunk。Report 和 News 为运行时计算实体，不持久化存储。

## 2. 存储引擎

| 项 | 说明 |
|----|------|
| 引擎 | JSON 文件（`json` 标准库） |
| 编码 | UTF-8，缩进 2 空格 |
| 读写模式 | RMW（全量读入 → 修改 → 全量写回） |
| 并发安全 | 无锁，不支持并发写入 |
| 文件布局 | `{DATA_DIR}/session_info.json` + `message_log.json` + `doc_chunks.json` |

> 文件命名对齐 08 §5.2 数据文件结构定义。

## 3. 实体定义 — Session

| 字段 | 类型 | 必填 | 约束 | 说明 |
|------|------|------|------|------|
| `session_id` | `string` (UUID) | 是 | **PK** | 会话唯一标识，对齐 09 `session_id` |
| `title` | `string` | 是 | ≤100 字符 | 默认"新会话"，首次问答自动重命名 |
| `created_at` | `string` (ISO-8601) | 是 | UTC+Z | 创建时间，对齐 09 `created_at` |
| `updated_at` | `string` (ISO-8601) | 是 | UTC+Z | 最后更新时间 |
| `query_count` | `integer` | 是 | ≥ 0 | 累计问答次数，对齐 09 `query_count` |
| `last_query` | `string` | 否 | — | 最后一次提问的原文，用于侧边栏显示摘要 |

## 4. 实体定义 — QARecord

| 字段 | 类型 | 必填 | 约束 | 说明 |
|------|------|------|------|------|
| `record_id` | `string` | 是 | **PK**，`rec_{ts}` | 基于时间戳生成，对齐 09 `record_id` |
| `session_id` | `string` | 是 | **FK** → Session | 所属会话 |
| `query` | `string` | 是 | 1–500 字符 | 用户提问原文，对齐 09 `query` |
| `answer` | `string` | 是 | — | LLM/Demo 返回的答案（Markdown），对齐 09 `answer` |
| `llm_used` | `boolean` | 是 | — | 是否使用真实 LLM，对齐 09 `llm_used` |
| `model` | `string\|null` | 是 | — | 模型标识，对齐 09 `model` |
| `response_time_ms` | `integer` | 是 | ≥ 0 | 响应耗时（ms），对齐 09 `response_time_ms` |
| `answer_source` | `string` | 是 | 枚举 `copaw`\|`bailian`\|`demo` | 答案来源，对齐 09 `answer_source` |
| `timestamp` | `string` (ISO-8601) | 是 | UTC+Z | 提问时间，对齐 09 `timestamp` |
| `references` | `array` | 否 | — | 引用出处列表，对齐 09 `references` |

| 字段 | 类型 | 必填 | 说明 |

| `chunk_id` | `string` | 是 | 文档块 ID，FK → DocChunk，对齐 09 `chunk_id` |
| `doc_title` | `string` | 是 | 源文档标题，对齐 09 `doc_title` |
| `page` | `integer` | 是 | 所在页码，对齐 09 `page` |
| `highlight_text` | `string` | 是 | 高亮原文段落，对齐 09 `highlight_text` |

## 5. Storage 方法清单

> 列出 Storage 类的全部公开方法。每个方法标注关联的 TC 编号（来自 `13`）。

### 5.1 会话管理

| 方法 | 签名 | 行为 | 关联 TC |
|------|------|------|---------|
| `create_session` | `(session_id: str, title: str) → dict` | 创建新会话，写入 `session_info.json`；`query_count` 初始化为 0，`created_at`/`updated_at` 设为当前 UTC 时间 | TC-M01-041 |
| `get_sessions` | `() → list[dict]` | 返回全部会话，按 `created_at` 倒序排列，对齐 09 §5 | TC-M01-042 |
| `delete_session` | `(session_id: str) → dict` | 删除指定会话 + **级联删除**该会话下全部 QARecord；返回 `{"deleted_session_id": ..., "deleted_records_count": ...}` | TC-M01-043 |
| `update_session` | `(session_id: str, updates: dict) → dict` | 按 `session_id` 查找会话，将 `updates` 字典中的键值合并写入（仅允许修改 `title`、`query_count`、`updated_at`），写回 `session_info.json`，返回更新后的完整会话对象；若 `session_id` 不存在则抛出 `KeyError` | TC-M01-046 |

### 5.2 问答记录管理

| 方法 | 签名 | 行为 | 关联 TC |
|------|------|------|---------|
| `add_record` | `(session_id: str, query: str, answer: str, llm_used: bool, model: str\|null, response_time_ms: int, answer_source: str, references: list\|None) → dict` | 写入一条 QARecord 到 `message_log.json`；同时调用 `update_session` 递增 `query_count` + 更新 `updated_at` | TC-M01-044 |
| `get_records_by_session` | `(session_id: str) → list[dict]` | 按 `session_id` 过滤，按 `timestamp` **正序**排列返回，对齐 09 §7 | TC-M01-045 |
| `delete_records_by_session` | `(session_id: str) → int` | 按 `session_id` 过滤并删除 `message_log.json` 中所有关联记录；返回实际删除的记录条数（`deleted_count`）；若无匹配记录返回 0 | TC-M01-047 |

### 5.3 文档块管理

| 方法 | 签名 | 行为 | 关联 TC |
|------|------|------|---------|
| `get_chunk_by_id` | `(chunk_id: str) → dict\|None` | 按 `chunk_id` 查找 DocChunk；未找到返回 `None`，对齐 09 §9 | TC-M01-048 |

## 6. 关键业务逻辑

**示例 — 首次问答自动命名**：

```python
if session["query_count"] == 1:
    session["title"] = query[:20] + ("..." if len(query) > 20 else "")
```

> **规则**：仅首次问答（query_count 从 0 → 1）触发，后续不再自动更名。

| 逻辑 | 触发条件 | 行为 |
|------|----------|------|
| 首次问答自动命名 | `query_count` 从 0 → 1 | 标题 = query 前 20 字 + `...`（超长时追加） |
| 级联删除 | 删除 Session | 同步删除所有关联 QARecord |
| 问答计数递增 | `add_record` 写入成功 | 目标 Session 的 `query_count` +1，`updated_at` 更新为当前 UTC 时间 |
| 会话排序 | `get_sessions` 调用 | 返回结果按 `created_at` 降序排列（最新在前），对齐 09 §5 AC-001-02 |
| 记录时间排序 | `get_records_by_session` 调用 | 返回结果按 `timestamp` 升序排列（最早在前），对齐 09 §7 AC-004-01 顺序一致 |
| 会话存在性校验 | `add_record` / `get_records_by_session` / `delete_records_by_session` 调用 | 若 `session_id` 在 `session_info.json` 中不存在，抛出 `KeyError`，由 Route 层映射为 404 `SESSION_NOT_FOUND`，对齐 09 §2 错误码清单 |
| 文档块存在性校验 | `get_chunk_by_id` 调用 | 若 `chunk_id` 在 `doc_chunks.json` 中不存在，返回 `None`，由 Route 层映射为 404 `DOC_CHUNK_NOT_FOUND`，对齐 09 §2 错误码清单 |

## 7. API 响应 DTO 概要

> 详细端点定义见 `09`。此处列出关键 DTO 与存储字段的映射关系。

| API 端点 | 核心 DTO 字段 | 额外注入 |
|----------|---------------|----------|
| `POST /ask` | answer, llm_used, model, response_time_ms, answer_source, references | `traceId` |
| `POST /sessions` | session_id, title, created_at, query_count | `traceId` |
| `GET /sessions` | sessions[]（session_id, title, created_at, query_count） | `traceId` |
| `DELETE /sessions/<id>` | deleted_session_id, deleted_records_count | `traceId`, `message` |
| `GET /sessions/<id>/records` | records[]（record_id, query, answer, timestamp, answer_source, references） | `traceId`, `session_id` |

| `GET /health` | status, components | — |
| `GET /reports` | reports[]（filename, companyName, stockCode, rating, targetPrice, currentPrice, upside） | — |
| `POST /reports/compare` | reports[]（companyName, rating, targetPrice, financials, coreViews, risks） | — |
| `GET /news` | news[]（id, title, summary, source, publishedAt, tags, priority）, sources[] | — |
