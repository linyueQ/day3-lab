> **04 的定位**：`03` 说了"为什么做"，04 翻译成"做什么、不做什么、多重要"。`05` 再拆成用户故事和验收标准。

## 1. 愿景与目标

> 把 `03` §1 的痛点翻译成正面目标。格式："构建 XXX，帮助用户在 XXX 内完成 XXX"。

构建投研问答助手，帮助投研分析师在 3 分钟内通过自然语言对话完成研报核心观点提取与交叉验证，免除手动翻阅数十页研报的低效劳动。同时提供研报横向对比与舆情实时监控能力，构建一站式投研信息枢纽。

| 03 痛点 | 04 目标 |
|---------|---------|
| 信息分散：研报、新闻、数据分布在多个系统，无统一问答入口 | 统一会话问答入口，分析师在一个界面内完成提问→获答→溯源全链路 |
| 阅读耗时：投研人员需大量时间阅读长篇研报提取核心观点 | 大模型自动摘要与结构化提炼，秒级返回评级、目标价等关键信息 |
| 研报对比难：多份研报的核心观点与数据难以横向对比 | 多维度研报对比（估值概览/财务对比/深度分析），支持 2~5 份研报同时横向对比 |
| 可信度挑战：AI 内容存在幻觉风险，无法快速核对原文 | 强制溯源机制，每条回答携带原文引用链接，点击即可定位研报原文段落 |
| 信息滞后：市场动态分散在多个新闻源，缺乏统一聚合入口 | 舆情监控面板，RSS 多源聚合 + 优先级自动评分，一屏掌握市场脉搏 |

## 2. 目标用户

> 把 `03` §2 的一句话展开成具体角色。思考：谁会用？用来做什么？多常用？

| 角色 | 使用目标 | 频率 |
|------|----------|------|
| 投研分析师 | 快速查询研报核心观点（评级、目标价、风险因素），辅助撰写研究报告 | 每天多次 |
| 投研部门主管 | 审阅分析师的问答记录，评估 AI 辅助效果与信息准确性 | 每周 2~3 次 |
| 运维监控人员 | 监控系统健康状态，确认 LLM 服务可用性与降级切换是否正常 | 每天 1 次 |

## 3. 核心业务场景

> 把 `03` §2 提案概要拆成 4~5 条用户路径。每条场景 = 一条用户操作动线。

| 编号 | 场景名 | 用户路径 |
|------|--------|----------|
| SC-01 | 新建会话并提问 | 用户点击"新会话" → 输入研报相关问题 → 获得 AI 结构化回答（含来源标识 + 引用标记）→ 回答自动落库 |
| SC-02 | 查看历史并追问 | 用户在侧边栏点击历史会话 → 主区域还原完整聊天上下文 → 在底部继续输入追问 → 系统结合历史上下文生成新回答 |
| SC-03 | 管理会话列表 | 用户浏览侧边栏会话列表（按时间倒序） → 点击删除某个过期会话 → 二次确认后列表刷新、关联记录清除 |
| SC-04 | 溯源核实原文 | 用户阅读 AI 回答时发现引用标记 `[1]` → 点击跳转至右侧 PDF 预览区 → 自动滚动定位至原文段落并高亮显示 → 核实数据准确性 |
| SC-05 | 降级与健康感知 | 运维人员访问 `/health` 确认系统状态 → 当 LLM 服务不可用时，前端自动展示"Demo 模式/服务受限"横幅 → 问答仍可返回离线演示回复 |
| SC-06 | 研报横向对比 | 用户在顶部导航切换至"研报对比"视图 → 选择 2~5 份研报 → 系统自动解析提取评级、目标价、财务预测等关键指标 → 分"估值概览/财务对比/深度分析"三个标签页呈现对比结果 |
| SC-07 | 舆情监控 | 用户在顶部导航切换至"舆情监控"视图 → 系统自动聚合 RSS 多源新闻 → 按 urgent/high/medium/normal 四级优先级展示 → 支持关键词搜索与按优先级标签页筛选 → 每 5 分钟自动轮询更新 |

## 4. 产品规则

> 把 `03` §4 成功标准 + §3 范围约束展开成可落地的规则。

| 编号 | 规则 | 来源 |
|------|------|------|
| R-01 | 每条提问必须绑定 `session_id`，无绑定时返回 400 `MISSING_SESSION_ID` | 03 §2 "会话式问答" |
| R-02 | `query` 长度 1～500 字符；为空返回 `EMPTY_QUERY`，超长返回 `INVALID_QUERY` | 03 §4 "P0 全部可测" |
| R-03 | 回答必须携带 `answer`、`llm_used`、`model`、`response_time_ms`、`answer_source` 字段；当基于原文提取时须携带 `references` 引用出处列表 | 03 §4 "核心信息溯源有效性" |
| R-04 | 三级降级：CoPaw → 百炼 → Demo 模式兜底；每级超时或异常时静默降级至下一级，三级全失败返回 500 `LLM_SERVICE_ERROR` | 03 §5 "外部 LLM 不可用" |
| R-05 | 会话首次问答（`query_count` 从 0→1）自动以 query 前 20 字重命名标题；新建时默认标题"新会话" | 03 §3 "会话管理" |
| R-06 | 删除会话时必须级联删除该会话下所有问答记录，操作不可逆 | 03 §3 "会话管理：创建、列表、删除" |
| R-07 | 问答结果的引用标记 `[n]` 必须可点击跳转至源 PDF 对应段落，引用链接准确率 100% | 03 §4 "核心信息溯源有效性" |
| R-08 | 助手仅提供信息辅助，绝对不生成或执行任何自动化交易指令 | 03 §3 "不做：交易指令" |
| R-09 | 研报对比支持 2~5 份同时选择；选择少于 2 份时"开始对比"按钮禁用 | 03 §3 "研报对比" |
| R-10 | 研报对比自动从 Markdown 研报中提取评级、目标价、当前价、上涨空间、营收/利润预测、核心观点、风险提示等多维度数据 | 03 §4 "研报对比可演示" |
| R-11 | 舆情监控新闻优先级分四级（urgent/high/medium/normal），基于中英文金融关键词 + 时效因素自动评分 | 03 §3 "舆情监控" |
| R-12 | 舆情监控数据 5 分钟内存缓存，支持手动强制刷新 | 03 §3 "舆情监控" |
| R-13 | 当所有 RSS 数据源不可用时，舆情监控自动降级至 Mock 数据 | 03 §5 "RSS 数据源不可用" |

## 5. 能力与优先级

> `03` §3 范围表写了"做什么"，04 按 P0/P1 排优先级。P0 = 必须实现，阻塞发布；P1 = 应当实现，不阻塞首版。

| 子能力 | 优先级 | 对应场景 | 对应 US |
|--------|--------|----------|---------|
| 会话管理（列表 / 新建 / 删除） | **P0** | SC-01, SC-03 | US-001 |
| 问答提交与三级降级 | **P0** | SC-01, SC-05 | US-002 |
| 历史记录还原与多轮追问 | **P0** | SC-02 | US-004 |
| 问答结果溯源（引用标记 + PDF 预览定位） | **P1** | SC-04 | US-003 |
| 健康检查与降级状态展示 | **P1** | SC-05 | US-005 |
| 研报横向对比（估值/财务/深度分析） | **P1** | SC-06 | US-006 |
| 舆情监控与新闻聚合 | **P1** | SC-07 | US-007 |

## 6. 验收与成功标准（PRD 层）

> 从 `03` §4 可度量标准出发，加上演示条件。

| 标准 | 度量 |
|------|------|
| 可演示 | 种子数据 ≥ 10 条下，主路径（新建→提问→历史→溯源）+ 能力状态 **3 分钟内**演示完毕 |
| 离线可用 | 未配置 LLM 密钥时系统仍返回离线演示回复（`answer_source=demo`），全链路不报错 |
| 响应效率 | 用户提交问答后，首字响应时间 < 3 秒，单次完整回答返回 < 15 秒（对齐 03 §4） |
| 溯源准确性 | 随机抽查 10 次数据问答，引用链接点击后 100% 准确跳转定位到源文档段落（对齐 03 §4） |
