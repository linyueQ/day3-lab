> **本文只做一件事**：按用户能理解的子场景拆成多条 **User Story**，并为每条写清 **REQ-ID**、**验收标准（AC）**。

## 字段说明 — 每条 UserStory 必须包含以下字段

| 字段 | 填写说明 | 数据来源 ← | 流向下游 → |
|------|----------|------------|-----------|
| **需求编号** | 唯一编号，如 REQ-FH-001 | ← `02` 采集编号 | → `14` 追踪矩阵首列 |
| **故事** | 作为【角色】，我希望【行为】，以便【价值】 | ← `03` 范围边界 | → `06` FSD 功能点 |
| **优先级** | P0（必须）/ P1（应该）/ P2（可选） | ← `03` §4 成功标准 | → `12` 里程碑排期 |
| **来源** | 对应 02/03 哪一条需求 | ← `02` 采集 · `03` 范围 | |
| **前置条件** | 触发该故事的前提状态 | ← 用户流程 | → `13` 测试前置 |
| **功能对齐** | 对应功能规格 + 非功能约束 | | → `06` FSD · `07` NFR |
| **设计对齐** | 对应架构、接口、数据、安全 | | → `08` 架构 · `09` API · `10` Data · `11` 安全 |
| **验证对齐** | 对应测试用例 + 追踪矩阵行 | | → `13` TC · `14` 追踪矩阵行 |
| **验收标准** | AC，每条可独立验证 | ← 故事 + `07` NFR | → `13` 测试用例依据 |

## US-001（P0）基金诊断 — 搜索与综合评级

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-FH-001 |
| **故事** | 作为个人投资者，我希望输入基金代码或名称后获取该基金的综合评级和多维度评分，以便快速判断基金整体质量 |
| **优先级** | P0 |
| **来源** | ← 02 S-01 痛点1 / 03 §3 范围1 |
| **前置条件** | 用户已进入基金诊断页面 |
| **功能对齐** | → 06 §基金诊断 · 07 §首屏性能 |
| **设计对齐** | → 08 §前端架构 · 09 GET /api/fund/diagnosis · 10 fund_info表 |
| **验证对齐** | → 13 TC-001 · 14 第1行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-001-01 | 输入基金代码（如 005827）并点击搜索后，页面展示基金名称、代码、类型、基金经理、基金公司、规模等基本信息 |
| AC-001-02 | 诊断结果展示综合评级（1-5星）和"优秀/良好/一般"等文字标签 |
| AC-001-03 | 展示5个维度的仪表盘评分：收益能力、风控能力、稳定性、择时能力、选股能力，每项0-100分 |
| AC-001-04 | 展示最新净值及日涨跌幅，涨为红色、跌为绿色 |
| AC-001-05 | 点击热门基金标签可快捷填入代码并触发诊断 |
