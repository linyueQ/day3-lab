> **本文只做一件事**：按用户能理解的子场景拆成多条 **User Story**，并为每条写清 **REQ-ID**、**验收标准（AC）**。

## 字段说明 — 每条 UserStory 必须包含以下字段

| 字段 | 填写说明 | 数据来源 ← | 流向下游 → |
|------|----------|------------|-----------|
| **需求编号** | 唯一编号，如 REQ-GOLD-001 | ← `02` 采集编号 | → `14` 追踪矩阵首列 |
| **故事** | 作为【角色】，我希望【行为】，以便【价值】 | ← `04` PRD 场景拆解 | → `06` FSD 功能点 |
| **优先级** | P0（必须）/ P1（应该）/ P2（可选） | ← `03` §4 成功标准 | → `12` 里程碑排期 |
| **来源** | 对应 03/04 哪一条需求 | ← `03` 范围 · `04` 场景 | |
| **前置条件** | 触发该故事的前提状态 | ← `04` 用户流程 | → `13` 测试前置 |
| **验收标准** | AC，每条可独立验证 | ← 故事 + `07` NFR | → `13` 测试用例依据 |

## US-001（P0）添加黄金记录

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-GOLD-001 |
| **故事** | 作为黄金投资爱好者，我希望录入每次黄金购买的重量、价格和渠道信息，以便统一管理所有黄金资产 |
| **优先级** | P0 |
| **来源** | ← 03 §3 / 04 SC-01 |
| **前置条件** | 用户已打开应用，当前账本已选定 |
| **功能对齐** | → 06 §2 录入页 · 07 §5 输入约束 |
| **设计对齐** | → 08 §2 · 09 POST /gold-records · 10 GoldRecord 表 |
| **验证对齐** | → 13 TC-GOLD-001~005 · 14 第1行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-001-01 | 用户输入重量（克，0.01~99999.99）、成交总价（元，>0）、购买渠道、日期后点击保存，系统返回 201 并创建记录 |
| AC-001-02 | 支持汇总模式（直接输入总价）和明细模式（单价×重量自动计算），模式切换后表单字段相应变化 |
| AC-001-03 | 重量或总价为空/超出范围时，前端提示错误，不允许提交 |
| AC-001-04 | 保存成功后自动跳转回账本列表页，资产总览数据自动更新 |

## US-002（P0）查看资产总览与记录列表

| 字段 | 内容 |
|------|------|
| **需求编号** | REQ-GOLD-002 |
| **故事** | 作为黄金投资爱好者，我希望在主页一眼看到总重量、购入成本、预估价值和预估收益，以便了解整体资产状况 |
| **优先级** | P0 |
| **来源** | ← 03 §3 / 04 SC-02 |
| **前置条件** | 账本中已有至少 1 条黄金记录 |
| **功能对齐** | → 06 §1 攒金主页 · 07 §1 延迟 SLO |
| **设计对齐** | → 08 §2 · 09 GET /gold-records · 10 GoldRecord 表 |
| **验证对齐** | → 13 TC-GOLD-010~013 · 14 第2行 |

| 验收标准 | 描述（每条可独立测试） |
|----------|----------------------|
| AC-002-01 | 主页顶部展示总重量(克)、购入总价(元)、预估价值(元)、预估收益(元) |
| AC-002-02 | 预估价值 = 总重量 × 最新金价，预估收益 = 预估价值 - 购入总价，收益为正显示绿色/红色 |
| AC-002-03 | 记录列表支持按购买渠道筛选、按日期/重量/价格排序 |
| AC-002-04 | 无记录时显示空状态图标 + "空空如也~" + "添加黄金"按钮 |

| **故事** | 作为黄金投资爱好者，我希望拍照自动识别黄金重量，以便减少手动输入的麻烦 |
| **优先级** | P1 |
| **来源** | ← 03 §3 / 04 SC-04 |
