## 1. 目标

本分工以**减少代码交叉、便于并行开发、降低集成测试风险**为第一原则。

核心策略：

1. **按模块边界拆人，不按接口数量拆人**
2. **共享文件只设一个 owner**
3. **前后端以 `09 API` 为唯一契约**
4. **集成测试按模块负责人分别验收，再由主集成人做总联调**

## 2. 人员与角色

| 成员 | 角色定位 | 主职责 |
|------|----------|--------|
| `b1` | 后端主干 / 集成人 | 后端骨架、核心 Skill 读写、共享存储、公共错误码与 traceId、总集成 |
| `b2` | 后端 ZIP / 文件域负责人 | ZIP 上传、解压校验、bundle 落盘、下载链路、安全清理 |
| `b3` | 后端搜索 / 互动域负责人 | LLM 草稿、智能搜索、点赞/收藏/评分、热度重算 |
| `f1` | 前端负责人 | 全部前端页面、API 对接、主题、页面状态、E2E |

## 3. 总体拆分原则

### 3.1 唯一 owner 原则

下列共享文件只允许一个人修改：

| 文件 / 模块 | 唯一 owner | 说明 |
|-------------|------------|------|
| `backend/app.py` | `b1` | 路由注册、应用初始化、错误处理、中间件 |
| `backend/storage.py` | `b1` | 所有 JSON / bundle / interaction 持久化接口 |
| `backend/tests/conftest.py` | `b1` | 后端测试基座、fixture、临时目录装配 |
| `frontend/src/api.js` | `f1` | 前端统一 API Client |
| `frontend/src/App.jsx` | `f1` | 前端路由与全局布局 |

### 3.2 契约冻结原则

- 前后端字段名、错误码、请求响应体**以 `09` 为准**
- 数据字段、聚合口径、热度公式**以 `10` 为准**
- 安全边界、限速、fallback 策略**以 `11` 为准**
- 任何人要改契约，先改文档，再改代码

### 3.3 禁止交叉原则

- `b2`、`b3` **不直接改** `backend/app.py`、`backend/storage.py`
- `b1` **不直接改** `frontend/src/**`
- `f1` **不直接改** `backend/**`
- 需要共享能力时，先由 `b1` 暴露方法签名，再由其他人接入

## 4. 按人拆分的开发任务

### 4.1 b1 — 后端主干 / Skill Core Owner

#### 负责范围

| 维度 | 内容 |
|------|------|
| 业务域 | Skill 基础读写、列表、详情、在线创建、分类、标签、Markdown 渲染接入 |
| 共享能力 | app 初始化、storage、错误码、traceId、测试基座 |
| 角色定位 | 其它后端模块的集成入口和最终合并人 |

#### 文件所有权

```text
backend/app.py
backend/storage.py
backend/blueprints/skill_bp.py
backend/services/skill_service.py
backend/services/md_render.py
backend/tests/conftest.py
backend/tests/test_skill_api.py
backend/tests/test_storage_core.py
```

#### 接口所有权

| 接口 | 说明 |
|------|------|
| `GET /api/v1/hub/skills` | 列表、分页、分类、标签、排序基础能力 |
| `GET /api/v1/hub/skills/<id>` | 详情、`view_count` 自增 |
| `POST /api/v1/hub/skills` | 在线创建 |
| `GET /api/v1/hub/skills/tags` | 标签 Top 20 |
| `GET /api/v1/hub/categories` | 分类白名单 + 动态计数 |

#### 必交付能力

1. 建好 Flask 应用骨架、蓝图注册、统一错误响应
2. 建好 `storage.py` 中所有共享方法签名
3. 打通 `skills.json / interactions.json / bundles/` 三类存储
4. 提供创建、列表、详情、标签、分类的基础可用版本
5. 负责最终集成其它后端分支

#### 对其它人的支持接口

`b1` 必须优先提供以下可调用方法，供 `b2`、`b3` 直接复用：

```text

| 安全域 | zip-slip、zip bomb、防残留脏文件 |

#### 文件所有权

```

#### 接口所有权

| 接口 | 说明 |
|------|------|
| `POST /api/v1/hub/skills/upload` | ZIP 上传发布 |
| `GET /api/v1/hub/skills/<id>/download` | ZIP 下载 + `download_count` 自增 |

#### 必交付能力

1. 完成 ZIP 大小、文件数、总解压大小、安全路径校验

#### 与他人的边界

- 不修改 `storage.py`，通过 `b1` 提供的方法写入 Skill 记录

## 6. 推荐开发顺序

### 6.1 第一阶段：先冻结共享底座

由 `b1` 优先完成：

1. `app.py` 基础骨架
2. `storage.py` 方法签名
3. 统一错误码和 `traceId`
4. `skills.json / interactions.json / bundles/` 目录结构

完成这一步后，`b2`、`b3` 就可以并行开发，不再抢共享文件。

### 6.2 第二阶段：三后端并行

| 成员 | 并行任务 |
|------|----------|
| `b1` | `GET /skills`、`GET /skills/<id>`、`POST /skills`、tags、categories |
| `b2` | upload / download / ZIP 安全链路 |
| `b3` | draft / smart-search / like / favorite / rate / ranking |

### 6.3 第三阶段：前端对接

`f1` 不必等后端全量完成，按以下顺序接：

1. 先对接 `GET /skills`、`GET /categories`
2. 再对接详情、创建、上传
3. 最后对接 draft、smart-search、互动接口

## 8. 合并与联调规则

### 8.1 分支建议

| 成员 | 分支名建议 |
|------|-----------|
| `b1` | `feature/b1-core` |
| `b2` | `feature/b2-upload` |
| `b3` | `feature/b3-search-interact` |
| `f1` | `feature/f1-frontend` |

### 8.2 合并顺序

1. `b1` 先合并共享底座
2. `b2`、`b3` 基于底座分别提 PR
3. `f1` 最后切到最新后端接口联调
4. 回归完成后由 `b1` 负责后端总收口，`f1` 负责前端总收口

### 8.3 唯一允许的共享变更

若必须改共享文件，只允许按下列方式进行：

1. 由需求方先在群里说明原因
2. 由 `b1` 统一改 `backend/app.py` / `backend/storage.py`
3. 其它后端只改自己名下文件

## 9. 风险与规避

| 风险 | 触发点 | 规避策略 |
|------|--------|----------|
| 三个后端同时改 `storage.py` | 共享存储冲突 | `storage.py` 只给 `b1` |
| 上传与详情都碰下载逻辑 | 路由归属不清 | 下载固定归 `b2` |
| 搜索与互动都依赖热度 | 公式口径不一致 | `ranking_service.py` 只给 `b3` |
| 前端等待后端太久 | 页面阻塞 | `f1` 先按 `09` mock 开发 |
| 集成时错误码不一致 | 文档与代码漂移 | 合并前逐项对照 `09` 和 `13` |

## 10. 最终建议

如果目标是**最快落地且最少冲突**，推荐按下面的一句话执行：

- `b1` 管后端主干和共享文件
- `b2` 管 ZIP 上传下载
- `b3` 管搜索、互动、热度
- `f1` 管全部前端

这样四个人基本可以做到**按目录并行开发**，只在接口联调和共享存储方法签名上发生一次对齐，最适合集成测试。
