# Spec 规格文档生成

## 概述

分两阶段生成完整 Spec 文档集：
1. **阶段一**：通过多轮对话确认，逐个生成上游文档（03→04→05→09）
2. **阶段二**：基于上游文档，批量生成下游文档（08→10→06→07→11→12→13→14）

## 文档依赖关系

```
阶段一：上游文档（多轮对话确认生成）
├── 03-立项提案与范围说明.md    # Why：背景、痛点、提案概要、范围边界
│       ↓
├── 04-产品需求说明.md         # What：愿景目标、用户角色、核心场景、产品规则（依赖 03）
│       ↓
├── 05-用户故事与验收标准.md    # What：US 编号、验收标准 AC（依赖 03、04）
│       ↓
└── 09-API接口规格.md          # How：端点定义、请求/响应 DTO、错误码（依赖 03、04、05）

阶段二：下游文档（批量生成）
├── 08-系统架构与技术选型.md    # 架构决策、技术选型理由
├── 10-数据模型与存储规格.md    # 实体定义、Storage 方法清单
├── 06-功能规格说明.md          # 前端 UI/UX 行为规格
├── 07-非功能需求与约束.md      # 性能、可用性、安全约束底线
├── 11-安全设计规格.md          # 安全目标、认证授权、数据安全
├── 12-实施计划与里程碑.md      # S 阶段、WBS 任务拆分
├── 13-测试策略与质量门禁.md    # 测试用例清单、门禁条件
└── 14-需求追踪矩阵.md          # 正向/反向追溯矩阵
```

## 阶段一：上游文档多轮对话生成

### 生成顺序

按依赖链顺序逐个生成，每个文档通过多轮对话确认：

**03 → 04 → 05 → 09**

---

### 1.1 生成 03-立项提案与范围说明

**目标**：确认项目背景、痛点、提案概要、范围边界。

**对话流程**：

```
第 1 轮：确认背景与痛点
- 询问：请描述项目背景和用户遇到的核心痛点（至少 2-3 条）
- 追问：这些痛点的来源是什么？（会议纪要/邮件/用户访谈）

第 2 轮：确认提案概要
- 询问：用 3-5 句话概括要做什么？
- 提示格式：在【XX应用】内交付【XX能力】

第 3 轮：确认范围边界
- 询问：明确"做什么"（范围内，3-4 条）
- 询问：明确"不做什么"（范围外，3-4 条）

第 4 轮：确认成功标准与风险
- 询问：写出 3-4 条可量化的验收标准
- 询问：列出 2-3 个主要风险及应对措施

第 5 轮：生成文档确认
- 展示完整文档预览
- 询问：是否确认生成？需要修改哪些内容？
- 确认后输出文件
```

**输出检查点**：
- [ ] 痛点 ≥ 2 条，含来源
- [ ] 提案概要清晰（3-5 句）
- [ ] 范围边界：做/不做各 ≥ 3 条
- [ ] 成功标准可量化
- [ ] 风险 ≥ 2 条，含应对

---

### 1.2 生成 04-产品需求说明

**前置条件**：03 已生成

**目标**：将 03 的"为什么做"翻译成"做什么"。

**对话流程**：

```
第 1 轮：确认愿景与目标
- 基于提取：将 03 §1 痛点翻译成正面目标
- 询问：格式"构建 XX，帮助用户在 XX 内完成 XX"
- 确认目标与痛点对应关系

第 2 轮：确认目标用户
- 基于 03 §2 提案概要展开
- 询问：具体角色、使用目标、使用频率

第 3 轮：确认核心业务场景
- 基于提取：将 03 提案概要拆成用户路径
- 询问：每个场景的编号、场景名、用户路径
- 确认场景与 05 用户故事对齐

第 4 轮：确认产品规则
- 基于提取：03 成功标准 + 范围约束
- 询问：每条规则的编号、规则内容、来源
- 确认规则可追溯到 03 章节

第 5 轮：确认能力优先级
- 基于 03 §3 范围表
- 询问：P0（必须实现）/P1（应当实现）划分
- 确认与场景对应

第 6 轮：生成文档确认
- 展示完整文档预览
- 询问：是否确认生成？需要修改哪些内容？
- 确认后输出文件
```

**输出检查点**：
- [ ] 愿景目标与 03 痛点对应
- [ ] 目标用户 ≥ 2 角色
- [ ] 核心场景 ≥ 3 条，含用户路径
- [ ] 产品规则 ≥ 4 条，含来源
- [ ] 能力优先级 P0/P1 划分明确

---

### 1.3 生成 05-用户故事与验收标准

**前置条件**：03、04 已生成

**目标**：将 04 场景拆成用户故事，每条写清验收标准。

**对话流程**：

```
第 1 轮：确认用户故事清单
- 基于 04 §3 核心业务场景
- 询问：每个场景对应一条 US
- 确认 US 编号格式：US-001、US-002...

第 2 轮：逐个确认 US 详情（多轮）
对于每个 US：
- 基于 04 场景填写故事格式：
  "作为【角色】，我希望【行为】，以便【价值】"
- 询问：优先级（P0/P1/P2）
- 询问：前置条件
- 确认来源：← 03 §X / 04 §X

第 3 轮：逐个确认验收标准（多轮）
对于每个 US：
- 基于 04 产品规则 + 场景描述
- 询问：每条 AC 可独立测试
- 确认 AC 编号格式：AC-{US}-{NN}

第 4 轮：确认需求追溯索引
- 汇总所有 US
- 确认下游文档引用预留

第 5 轮：生成文档确认
- 展示完整文档预览
- 询问：是否确认生成？需要修改哪些内容？
- 确认后输出文件
```

**输出检查点**：
- [ ] US 数量与 04 场景对应
- [ ] 每个 US 含：编号、故事、优先级、来源、前置条件
- [ ] 每个 US 含验收标准 AC ≥ 2 条
- [ ] AC 可独立测试

---

### 1.4 生成 09-API接口规格

**前置条件**：03、04、05 已生成

**目标**：定义所有 API 端点的契约。

**对话流程**：

```
第 1 轮：确认端点总览
- 基于 05 US 提取功能需求
- 询问：需要哪些 API 端点？
- 确认：端点路径、HTTP 方法、功能、对应 US
- 输出端点总览表

第 2 轮：确认统一响应规范
- 询问：成功响应格式（traceId + 业务字段）
- 询问：错误响应格式（error.code + message + details）
- 确认错误码清单

第 3 轮：逐个确认端点详情（多轮）
对于每个端点：
- 基于 US 验收标准确定请求参数
- 询问：路径参数、查询参数、请求体字段
- 确认：成功响应字段（类型、必填、说明）
- 提供响应示例

第 4 轮：确认参数校验规则
- 汇总所有字段的校验规则
- 确认：失败 HTTP、error.code

第 5 轮：确认 US 映射关系
- 汇总 US → API 端点映射
- 确保每个 US 有对应 API

第 6 轮：生成文档确认
- 展示完整文档预览
- 询问：是否确认生成？需要修改哪些内容？
- 确认后输出文件
```

**输出检查点**：
- [ ] 端点总览表完整，对应 US
- [ ] 统一响应规范定义
- [ ] 错误码清单 ≥ 5 条
- [ ] 每个端点含：请求参数、成功响应、错误响应、示例
- [ ] 参数校验规则汇总

---

### 阶段一完成检查

上游文档全部生成后，确认：

| 文档 | 核心内容检查点 | 状态 |
|------|----------------|------|
| 03 | 痛点 ≥ 2 条、提案概要、范围边界 | ☐ |
| 04 | 愿景目标、目标用户、核心场景 ≥ 3 条、产品规则 ≥ 4 条 | ☐ |
| 05 | US 编号、故事、验收标准 AC | ☐ |
| 09 | 端点列表、请求/响应 DTO、错误码清单 | ☐ |

**全部确认后进入阶段二。**

## 阶段二：下游文档批量生成

### 生成顺序

按依赖链顺序批量生成：

**08 → 10 → 06 → 07 → 11 → 12 → 13 → 14**

### 执行步骤

```
对于每个下游文档：
1. 读取对应模板
2. 识别【填写】【请填写】【请补充】标记
3. 从上游文档（03、04、05、09）提取相关信息
4. 填充内容，保留已有内容不变
5. 输出到指定目录
```

### 交叉引用检查

确保生成的文档中：
- 章节引用正确（如 `← 03 §1`、`→ 06 §2`）
- REQ 编号与 US 编号对应
- TC 编号与 REQ 编号对应

## 填充规则（阶段二适用）

### 从上游文档提取信息

| 待填区域 | 信息来源 | 提取方法 |
|----------|----------|----------|
| 模块编号 | 04 PRD | `模块编号` 字段 |
| 模块名称 | 04 PRD | `模块名称` 字段 |
| 技术栈 | 03 提案 | `技术栈` 字段 |
| 功能需求 | 05 US | US 故事内容 |
| API 端点 | 09 API | 端点列表 |
| 数据实体 | 09 API | 响应 DTO 字段 |
| 产品规则 | 04 PRD | §4 产品规则表 |
| 性能指标 | 04 PRD | §6 验收标准 |
| 风险项 | 03 提案 | §5 风险表 |

### 章节引用格式

- 上游引用：`← 03 §1`（来源是 03 文档第 1 节）
- 下游引用：`→ 06 §2`（下游是 06 文档第 2 节）
- 对齐引用：`对齐 09 §3.1`（与 09 文档第 3.1 节对齐）

### 编号命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 需求编号 | REQ-{模块}-{NNN} | REQ-M1QA-001 |
| 用户故事 | US-{NNN} | US-001 |
| 验收标准 | AC-{US}-{NN} | AC-001-01 |
| 产品规则 | R-{NN} | R-01 |
| 测试用例 | TC-{模块}-{NNN} | TC-RA-011 |

## 输出规范

### 文件命名

与模板保持一致，不修改文件名：
- `06-功能规格说明.md`
- `07-非功能需求与约束.md`
- ...

### 版本信息

每份文档末尾添加版本记录：

```markdown
| 版本 | 日期 | 说明 |
|------|------|------|
| v0.1 | {生成日期} | 基于 03 提案、04 PRD、05 用户故事、09 API 首版生成 |
```

### 不修改原则

- 保留模板中已有内容不变
- 仅填充标记为待填的区域
- 不删除模板中的示例或引导语

## 示例：多轮对话生成 03-立项提案

**第 1 轮对话**：
```
Agent: 请描述项目背景和用户遇到的核心痛点（至少 2-3 条）

User: 
1. 研报信息提取与比对困难，投研人员需要手动翻阅多份研报
2. 研报 AI 问答缺乏支撑，无法基于研报内容进行智能问答

Agent: 这些痛点的来源是什么？

User: 会议纪要、用户访谈
```

**第 2 轮对话**：
```
Agent: 用 3-5 句话概括要做什么？格式建议：在【XX应用】内交付【XX能力】

User: 在【研报助手】应用中交付研报信息提取与比对功能、研报AI问答功能。
```

**第 3 轮对话**：
```
Agent: 请明确"做什么"（范围内，3-4 条）和"不做什么"（范围外，3-4 条）

User:
做：研报信息提取与比对功能、研报AI问答
不做：舆情监控、股票数据查看
```

**生成结果**：
```markdown
## 1. 背景与问题

| 痛点 | 来源 |
|------|------|
| 研报信息提取与比对困难 | 会议纪要、用户访谈 |
| 研报AI问答缺乏支撑 | 会议纪要 |

## 2. 提案概要

在【研报助手】应用中交付研报信息提取与比对功能、研报AI问答功能。

## 3. 范围边界

| 做（范围内） | 不做（范围外） |
|-------------|---------------|
| 研报信息提取与比对功能、研报AI问答 | 舆情监控、股票数据查看 |
```
